## Root cause — cross-module `PATH`-mutation race in the new tests

The watcher tests added by WO-0027 (`test_watchman_detection`, `test_auto_select_backend_returns_notify_when_watchman_absent`, `test_auto_select_backend_returns_watchman_when_available_and_above_threshold`, `test_auto_select_backend_returns_notify_when_below_threshold`) mutate the process-wide `PATH` environment variable via `std::env::set_var("PATH", <empty-tempdir>)` and then restore it on `Drop` via the `RestorePath` RAII guard.

`std::env::set_var` is **process-global**, not thread-local. The module-level `ENV_GUARD` (`crates/ucil-daemon/src/watcher.rs:1076`) only serialises *watcher-to-watcher* PATH mutations. It does **not** prevent leakage into concurrent tests in other modules of the same test binary.

When `cargo test -p ucil-daemon --lib` runs (the harness path used by `cargo llvm-cov`), all tests execute as threads inside one process. While a watcher PATH-mutation test holds the `ENV_GUARD` and sets PATH to an empty tempdir, concurrent `session_manager::tests::*` tests invoke `git` via `tokio::process::Command::new("git")` (session_manager.rs:291/311/339). With the empty PATH, `git` is not resolvable and the spawn fails with `Io(Os { code: 2, kind: NotFound, message: "No such file or directory" })`.

### Why the executor's local check missed this

The executor verified under `cargo nextest run -p ucil-daemon --no-fail-fast` (62/62 passing). Nextest runs each test in its **own process** via `fork+exec`, so the `PATH` mutation in a watcher test cannot leak to a session_manager test in a different process. This is *not* the path the coverage gate takes — `cargo llvm-cov` drives `cargo test` directly, which uses libtest's thread-per-test-within-one-process model. The gate has been the canonical catch for exactly this kind of regression since Phase 0.

## Not a harness flake

WO-0026 retry-1/retry-2 saw a superficially similar "cargo llvm-cov report errored" symptom, but the cause there was leftover `.profraw` files in a stale `target/` from a previous partial run. That was fixed by `rm -rf target/` in retry 3 (commit `2b6c066` prunes zero-byte profraws as defense-in-depth). On WO-0027 the repro already starts from `rm -rf target/` and the failure persists; the corrupted profraw is produced **by this run**, because `cargo test` aborts when one of its threads panics and the in-flight profraw buffer for that thread is truncated rather than closed cleanly.

## Suggested fix (not applied by verifier — for executor or root-cause-finder)

Options, roughly by invasiveness:

1. **Isolate the PATH-mutating tests into a separate process.** The cleanest fix is to spawn the watchman-detection check as a short test binary under `tests/` and shell out with `Command::new(env!("CARGO_BIN_EXE_..."))` so the PATH mutation never lands in the shared cargo-test process. This does not change the frozen F03 selector because the test name is the selector — but it moves from `#[cfg(test)]` in `src/watcher.rs` to `tests/watchman_detection.rs`, which would be a visible structural change and may need an ADR.
2. **Side-step `PATH` entirely.** `which::which_in` lets the caller pass an explicit search path. Rewrite `detect_watchman()` to take an optional `&OsStr` PATH override (or read `UCIL_WATCHMAN_PATH_OVERRIDE` if set), and have `test_watchman_detection` exercise that override instead of mutating the process env. The public API stays backward-compatible because the override parameter is additive.
3. **Serialise globally across the whole daemon test binary.** Promote `ENV_GUARD` to a workspace- or daemon-wide `parking_lot::Mutex` guard that every daemon test that touches env *or* spawns an external binary must take. This is broad and touches files in `forbidden_paths` (`session_manager.rs`, `plugin_manager.rs`) — likely needs an ADR and scope expansion.

Option (2) looks smallest and cheapest; it keeps the test module-local and the frozen selector intact, and the `which_in` API is documented and stable in the `which` 7.x series already in the workspace deps.
