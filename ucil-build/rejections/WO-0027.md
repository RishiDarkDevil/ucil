## Root cause — cross-module `PATH`-mutation race (unchanged from retry 1 / retry 2)

The four WO-0027 watcher tests that mutate the process-wide `PATH`
environment variable —

- `watcher::test_watchman_detection` (the F03 oracle)
- `watcher::test_auto_select_backend_returns_notify_when_watchman_absent`
- `watcher::test_auto_select_backend_returns_watchman_when_available_and_above_threshold`
- `watcher::test_auto_select_backend_returns_notify_when_below_threshold`

— do so via `std::env::set_var("PATH", <empty-tempdir>)` behind a
module-level `static ENV_GUARD: std::sync::Mutex<()>` + `RestorePath`
RAII guard (`crates/ucil-daemon/src/watcher.rs:1076-1108`). The guard
serialises *watcher-to-watcher* PATH mutations correctly, **but**
`std::env::set_var` is process-global, not thread-local, and the
guard does not cover concurrent tests in other modules of the same
test binary.

When `cargo test -p ucil-daemon --lib` runs (the harness path
`cargo llvm-cov` drives), every test executes as a thread inside one
libtest process. While a watcher test holds `ENV_GUARD` and
`PATH=<empty tempdir>`, concurrent `session_manager::tests::*` tests
invoke `tokio::process::Command::new("git")` from
`session_manager.rs:291` / `:311` / `:339`. `git` is unresolvable in
the empty PATH and the spawn fails with `Io(NotFound)`, panicking
4 tests.

`cargo nextest` (the executor's local check path) runs each test in a
separate child process via `fork+exec`, so the PATH mutation cannot
leak between tests — that is why the executor's `62/62` nextest green
signal missed this for three iterations. The coverage gate has been
the canonical catch for this failure mode since Phase 0.

## Suggested fix (for planner / root-cause-finder — verifier cannot apply)

**Option 2 (smallest blast radius) — side-step `PATH` entirely.**

`which::which_in` (already available in the workspace's `which` 7.x
series — no new dep) lets the caller pass an explicit search path.
Rewrite `detect_watchman()` to accept an optional `&OsStr` PATH
override (additive, backward-compatible), and have
`test_watchman_detection` / the three `auto_select_backend_*` tests
exercise that override instead of mutating the process env. Public
API stays backward-compatible; frozen F03 selector name stays.

Concretely:

```rust
pub fn detect_watchman() -> Option<WatchmanCapability> {
    detect_watchman_in(std::env::var_os("PATH"))
}

pub fn detect_watchman_in(search_path: Option<std::ffi::OsString>) -> Option<WatchmanCapability> {
    let cwd = std::env::current_dir().ok()?;
    match which::which_in("watchman", search_path, &cwd) {
        Ok(binary) => Some(WatchmanCapability { binary }),
        Err(_) => None,
    }
}
```

Then update the four `PATH`-mutating tests to call `detect_watchman_in(Some(<shim-dir>.into()))`
(or build a custom PATH OsString and pass it) instead of
`std::env::set_var("PATH", …)`. The `ENV_GUARD` static and
`RestorePath` RAII guard can be deleted along with the PATH mutations.

**Alternative options** (from retry 1):
- Option 1 — isolate PATH-mutating tests into a separate `tests/`
  integration binary (respect DEC-0005 — module-root placement in the
  new binary still applies).
- Option 3 — `scripts/verify/coverage-gate.sh` could pass `--test-threads=1`
  to `cargo test` under coverage, eliminating the race; this is a
  harness-scope change, ADR-worthy, and leaves the underlying bug
  unfixed for future regressions.

Option 2 is strictly additive to the public API (no breaking change),
keeps the frozen F03 selector name, and removes the root-cause rather
than papering over it. Recommend Option 2 for the next executor
attempt.
