## Failed criteria

### Criterion 1 — nextest regex mismatch

**Criterion** (`acceptance_criteria[0]`):

```
cd $REPO_ROOT && cargo nextest run --test test_lsp_bridge --no-fail-fast 2>&1 \
  | tee /tmp/nextest-WO-0038.log \
  && grep -E 'tests run: ([5-9]|[1-9][0-9]+).*[0-9]+ passed' \
          /tmp/nextest-WO-0038.log > /dev/null
```

**Observed**: `cargo nextest run --test test_lsp_bridge --no-fail-fast` exits
0 with all 5 tests green. The summary line emitted by nextest v0.9.x is:

```
     Summary [   0.005s] 5 tests run: 5 passed, 0 skipped
```

The trailing `grep -E` regex then fails with exit 1 because after matching
`tests run: 5`, it requires an intervening `.*[0-9]+ passed` — i.e. a
separate `<digits> passed` token later in the line. In the passing-path
nextest summary, the number right after `tests run: ` IS the same number
that precedes ` passed`, so there is no second ` [0-9]+ passed` occurrence
for the regex to bite on. The non-matching `0 skipped` at the end tempts the
greedy `.*` to consume to the end, but backtracking never produces a
`[0-9]+ passed` that sits AFTER the captured `[5-9]` group and BEFORE the
end of the summary line.

Manual confirmation:

```
$ grep -E 'tests run: ([5-9]|[1-9][0-9]+).* passed' /tmp/nextest-WO-0038.log
     Summary [   0.005s] 5 tests run: 5 passed, 0 skipped
$ grep -E 'tests run: ([5-9]|[1-9][0-9]+).*[0-9]+ passed' /tmp/nextest-WO-0038.log
(no match, exit 1)
```

Removing the stray `[0-9]+` token after `.*` would make the regex match. As
written, the criterion is unreachable for all-green nextest output and so
fails every time regardless of executor correctness.

**Likely cause**: planner-side bug in `acceptance_criteria[0]` of
`ucil-build/work-orders/0038-lsp-bridge-integration-test-suite.json`. The
pattern `tests run: ([5-9]|[1-9][0-9]+).*[0-9]+ passed` over-constrains the
nextest summary. Either:
- The regex should be `tests run: ([5-9]|[1-9][0-9]+) passed` (nextest
  v0.9 format, which is what the binary actually emits), or
- The regex should be `([5-9]|[1-9][0-9]+) tests run: [0-9]+ passed` to
  anchor against the leading `<count> tests run` token.

This cannot be fixed by the executor (modifying the work-order's
`acceptance_criteria` is out of scope and changes the oracle). A planner
amendment or ADR is required.
