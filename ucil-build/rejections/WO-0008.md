#### Why this matters

The test `test_session_state_tracking` exercises each new method
exactly once along a single input. A mutation score of 41% means the
test suite cannot distinguish the correct `purge_expired` from trivial
arithmetic substitutions (`-→+`, `>→==`, `>→>=`). P1-W4-F07's contract
asserts TTL-expiry semantics; without boundary testing the code is not
meaningfully covered.

The minimum-viable fix (to reach ≥70%) is additional test cases — **no
source change is required** for the state-tracking logic itself:

1. Assert the default-TTL branch: create a session without calling
   `set_ttl`, call `purge_expired` at `created_at + DEFAULT_TTL_SECS +
   1` and confirm removal. This kills mutants #2 and #3.
2. Assert the `>` boundary: create a session with `set_ttl(10)`, call
   `purge_expired(created_at + 10)` (== `expires_at`), confirming it
   is NOT purged; then call `purge_expired(created_at + 11)` and
   confirm it IS purged. This kills mutants #5 and #6.
3. Assert the count with multiple sessions: create 3 sessions, set
   TTLs so 2 expire and 1 does not; confirm `purge_expired` returns
   2 and the survivor remains. This kills mutant #7.

Mutant #1 (`main.rs`) and mutant #4 (`detect_branch`) pre-date this WO
and are best addressed in a follow-up micro-WO; they contribute to the
crate-level score but are not the primary blocker.
