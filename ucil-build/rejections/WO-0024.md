since WO-0009, commit `5c2739a`):

```
crates/ucil-core/src/incremental.rs:5:
  //! ([`FileRevision`]) to two tracked query functions ([`symbol_count`] and
crates/ucil-core/src/incremental.rs:6:
```

(`symbol_count` and `dependent_metric`), so `rustdoc` cannot disambiguate
the unadorned `[`symbol_count`]` and `[`dependent_metric`]` references.
`rustdoc`'s own suggested fix is a 4-character edit per link (append `()`
for the function form or prefix `struct@` for the struct form).

## Suspected cause

Planner oversight: when WO-0024's acceptance gate was authored the planner
did not preflight `cargo doc -p ucil-core --no-deps` on `origin/main`. Had
it run the criterion on `main` first, it would have discovered the pre-existing
`symbol_count` / `dependent_metric` ambiguity (introduced silently by
WO-0009 because WO-0009's own gate did not include `cargo doc`) and either:

1. Emitted a one-line prerequisite WO to fix `incremental.rs` first (the
   orthodox Bucket-D path); or
2. Excluded `incremental.rs` from the `cargo doc` check with a
   `rustdoc::broken_intra_doc_links` allow on that file.

The executor correctly identified the issue in the ready-for-review marker
but is out of scope to fix (WO-0024 `scope_in` only names `Cargo.toml`,
`crates/ucil-core/Cargo.toml`, `crates/ucil-core/src/knowledge_graph.rs`,
and `crates/ucil-core/src/lib.rs`; `incremental.rs` is not listed, and
`scope_out` forbids a new ADR absent a genuine spec ambiguity or build
failure — `cargo build` is green, only `cargo doc` fails).
