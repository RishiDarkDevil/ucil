## Suspected cause

**False positive in `scripts/reality-check.sh` commit-selection logic**, not a genuine fake-green test.

The script finds the most-recent commit tagged `Feature: P0-W1-F02` that changed a source file:

```
e5ecae2 feat(core): wire lib.rs — declare modules and re-export public symbols  ← selected
ea983dd feat(core): implement types.rs — 7 serde domain types                   ← skipped
d6eeb7c build(core): add serde, rusqlite, opentelemetry workspace deps           ← skipped
```

It selects `e5ecae2` (lib.rs wiring commit). When lib.rs is reverted to remove `pub mod types;`,
`cargo test -p ucil-core types::` finds **0 matching tests** (the module is not declared) and
exits 0. The script interprets exit 0 with 0 tests as "acceptance tests passed" → fake-green.

The actual implementation (`crates/ucil-core/src/types.rs`) was not stashed. If the script had
selected `ea983dd` (the types.rs commit) and stashed that file, the `types::` tests would have
correctly failed.

Note: this specific failure mode — a multi-commit feature where the wiring commit is newer than
the implementation commit — was NOT covered by the fix in commit `92e8082`. That fix handled
administrative commits (ready-for-review markers) with no source files; it did not handle the
case where the most-recent source commit is a wiring commit for an already-separately-committed
implementation.
