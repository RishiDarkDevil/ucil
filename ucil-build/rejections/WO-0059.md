## Suspected cause

The WO scope_in[8] prescribed only 5 sub-assertions in the frozen test
(`SA1`–`SA5`), all on the happy path. The standing crate-level coverage
floor of 85% line / 75% branch was implicit (verifier-instructions step 7
default), not pinned in the WO's `acceptance` block. The pre-existing
`onnx_inference.rs` had only 10 uncovered lines (mostly the
`MissingInput` / `MissingOutput` defensive branches in
`OnnxSession::infer`, since the minimal-model test doesn't trigger them
either) and squeaked through at 87.80% — but that ratio relied on a
small absolute file size (82 lines). The new `models.rs` is 113 lines
with seven `CodeRankEmbedError` variants, six of which are absolutely
unreachable from a single happy-path test. Adding the new file
mechanically pulls the crate average below the 85% line floor.

The clean fix is to add **one or two negative-path tests** alongside the
frozen acceptance test. Two illustrative tests (executor's choice on
exact shape):

1. **`load_returns_missing_model_file_when_model_onnx_absent`** — pass a
   tempdir containing only `tokenizer.json` (or an empty dir) →
   asserts `Err(CodeRankEmbedError::MissingModelFile { ... })`. Covers
   line 261 and (if extended to a second `tokenizer.json`-absent case)
   lines 267-269.
2. **`embed_returns_dimension_mismatch_when_output_named_wrong`** — would
   need either a synthetic `[1, 769]`-output ONNX fixture OR a unit-level
   refactor that makes the `outputs.get("sentence_embedding")` call
   testable in isolation (e.g. extract a `pool_and_normalise(raw: &[f32])
   -> Result<Vec<f32>, _>` helper that the test calls directly with a
   wrong-length slice). The first option requires committing a tiny new
   ONNX fixture; the second is purer Rust.

A third (likely simplest) option is a `tokenizer_from_file_error_when_path_invalid`
test (covers lines 273-274) using a tempdir with a malformed `tokenizer.json`.

Adding any two of these, ideally three, raises `models.rs` from 74.34%
to ≥85% and the crate from 80% to ≥85%.

These additions stay within the WO's `scope_in` (the test file is
`crates/ucil-embeddings/src/models.rs` — the same file as the frozen
test, allowed). They do NOT need to live in `crates/ucil-embeddings/tests/`
(which is in the WO's `forbidden_paths`) — they can be additional
`#[test]` functions in `models.rs` next to the existing
`test_coderankembed_inference`.
