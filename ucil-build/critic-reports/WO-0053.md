### Soft notes / warnings

1. **Branch recreation pattern.** The anti-laziness contract forbids
   `git commit --amend` after push and `git push --force[-with-lease]`.
   The executor took the rule-compliant route documented in the
   ready-for-review marker (`0053-ready-for-review.md:62-77`):
   `git push origin --delete feat/WO-0053-lancedb-per-branch` (a
   delete-ref, not a force-push), then `git branch -D` locally, then
   re-create from `main` with shorter commit subjects. The result is
   that the 9 retry-1 shas (`e8b5a88…7b0932a`) are no longer reachable
   from any ref — only `git reflog` retains them. This is mechanically
   compliant: no force-push signature, no `--amend` after push, the
   branch was published once with the new sha chain and never
   rewritten on the remote. The previous critic report (`edd8ce6` on
   `main`) already discusses this trade-off in its AC22 "fix path"
   block; the executor picked option (b) (orphan re-creation). No
   action required.

2. **`PROTOC` build dependency.** Several transitive deps under
   `lancedb 0.16` (notably `lance-encoding` and `lance-file`) require
   the `protoc` binary on the build host — `cargo build` will fail at
   `lance-encoding-build.rs` if the system has neither
   `apt-get install protobuf-compiler` nor `PROTOC=…/protoc`
   exported. The retry-1 ready-for-review marker called this out
   (line 53 of the *prior* marker); the retry-2 marker omits the
   prerequisite section. Verifier sessions that spawn fresh shells
   without inheriting the host's `PROTOC` env will fail AC09 / AC11 /
   AC12 / AC13 / AC15 at the `lance-*-build.rs` step. Suggest the
   verifier check
   `command -v protoc || echo "PROTOC required"` before running cargo
   and either install `protobuf-compiler` or export `PROTOC=…`.
   Forward-looking soft warning, not a defect of the diff.

3. **One commit (`faacb88`) is dominated by `Cargo.lock` churn.**
   ~4614 lines of lockfile delta from adding `lancedb 0.16` cannot be
   atomically split — adding the workspace dep necessarily updates
   the lockfile in the same commit. The 32-line `Cargo.toml` and
   9-line `crates/ucil-daemon/Cargo.toml` deltas are under the soft
   target. Acceptable per `.claude/rules/commit-style.md`'s "big
   refactors split into review-friendly chunks" carve-out. No action
   required.

4. **AC17's regex remains a footgun for future tooling drift.** Even
   though the executor fixed the 6 doctest lines, a future PR that
   adds `///[space]assert!(…)` anywhere in `branch_manager.rs` will
   resurrect the failure. `.claude/rules/rust-style.md` requires
   `# Examples` blocks on every public item, and operator-readable
   assertion checks are the natural pattern. Suggest a planner micro-
   ADR to either (a) tighten the regex (`^[[:space:]]*//[^/]`,
   excluding `///`), or (b) drop AC17 in favour of the existing
   master `! grep` shape in `scripts/gate/`. Out of critic scope to
   write the ADR — flagging for the planner. Forward-looking, not
   blocking.

5. **`tempfile.workspace = true` listed twice in
   `crates/ucil-daemon/Cargo.toml`.** Lines 77 (under `[dependencies]`)
   and 80 (under `[dev-dependencies]`). Cargo permits this without
   warning (the `[dev-dependencies]` entry overrides for test/bench
   builds), but a future cleanup could move `tempfile` to
   `[dev-dependencies]` only — the production `BranchManager` does
   not depend on it. Pre-existing in the crate's manifest (visible in
   the diff because the new lancedb / arrow lines land alphabetically
   adjacent), not introduced by this WO. Out-of-scope soft note.
