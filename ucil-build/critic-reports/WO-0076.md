### Warnings (should fix before gate — non-blocking)

1. **`unsafe std::env::set_var("SEMGREP_PATH", ...)` without
   serial-test mutex protection or `--test-threads=1` enforcement.**
   `crates/ucil-daemon/tests/g7_plugin_manifests.rs:342-344` calls
   `unsafe { std::env::set_var(...) }` from the body of
   `semgrep_manifest_health_check`. The SAFETY comment at
   `tests/g7_plugin_manifests.rs:335-341` claims the test is run
   single-threaded "per `--test-threads=1` (WO-0076 verifier protocol —
   single-threaded tests)", but I cannot find that protocol enforced
   anywhere in the diff:
   - `scripts/verify/P3-W11-F02.sh:90-91` invokes
     `cargo test -p ucil-daemon --test g7_plugin_manifests
     g7_plugin_manifests::eslint_manifest_health_check` — no
     `--test-threads=1`.
   - `scripts/verify/P3-W11-F04.sh:132-133` invokes the analogous
     selector — no `--test-threads=1`.
   - `.cargo/config.toml` (managed by `scripts/setup-build-cache.sh`)
     has no `test-threads` setting.
   - The test file does NOT pull in `serial_test` or any
     synchronization primitive.

   The single-test selectors used by the verify scripts incidentally
   sidestep the race (only one test runs per `cargo test` invocation),
   but a verifier or future caller running the WHOLE file
   (`cargo test -p ucil-daemon --test g7_plugin_manifests` with no
   selector — as the RFR's "Live integration verification" block at
   `ucil-build/work-orders/0076-ready-for-review.md:197` does, where
   the executor explicitly added `-- --test-threads=1`) on a
   multi-core machine WILL execute both `#[tokio::test]` async fns in
   parallel by default. Rust 2024's `set_var` is `unsafe` precisely
   because concurrent set_var/getenv across threads is UB; the
   eslint test does not read `SEMGREP_PATH` explicitly but the
   tokio runtime + `tokio::process::Command::new("npx")` spawn paths
   internally consult libc `getenv` while resolving PATH, opening a
   real (small) race window.

   Mitigation options the executor or the verifier may apply
   pre-merge:
   - Add `serial_test` as a `dev-dependency` and annotate both
     `#[tokio::test]` fns with `#[serial(env_semgrep)]`.
   - Wrap the `set_var` in a `std::sync::Mutex` lazy-initialised
     `OnceLock<()>`.
   - Hard-pin `--test-threads=1` in both verify scripts.
   - Use `Command::env("SEMGREP_PATH", &semgrep_path)` directly on
     the spawned child via the public `PluginManager::spawn` (or
     `health_check_with_timeout`) API surface rather than mutating
     the parent's process environment — this is the cleanest
     solution and avoids the unsafe block entirely. (The
     `PluginManager` source on this branch already reads the
     manifest's `transport.command` + `transport.args`, but not a
     manifest-level `env` table — which is a daemon-side change
     that's out of scope for this WO. So this option is a
     follow-up, not a same-WO fix.)

   Soft warning — not gate-blocking because the verify scripts'
   single-selector cargo invocations avoid the race in practice and
   the unsafe block is documented (not silent). But the SAFETY
   reasoning is wrong as written: there is no
   `--test-threads=1`-enforcing protocol in the diff that the
   comment is appealing to.

2. **`unsafe` block without ADR justification.**
   `.claude/rules/rust-style.md` §Unsafe states:
   `Forbidden by default. Any unsafe block requires an ADR
   justification and // SAFETY: comment explaining invariants.`
   The test file at `crates/ucil-daemon/tests/g7_plugin_manifests.rs:342`
   has the `// SAFETY:` comment but does NOT cite any ADR. This is
   the first `unsafe` block introduced by any of the
   `tests/g{N}_plugin_manifests.rs` peer files (verified via
   `Grep("unsafe", crates/ucil-daemon/tests/")` which returned only
   `tests/g7_plugin_manifests.rs`). Per the rule, an ADR
   (e.g. `ucil-build/decisions/DEC-NNNN-test-env-var-mutation.md`)
   should accompany the introduction. Soft warning — the rust-style
   rule is project policy, not an oracle invariant; verifier may
   accept with a follow-up ADR ticket. Tied to Warning 1's
   suggested cleaner fix (env-on-Command), which would remove the
   unsafe block entirely.
