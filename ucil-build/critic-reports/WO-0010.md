### Warnings (should fix before gate, but not verifier-blocking)

1. **Large single-commit module file — `98b755e` @ 674 insertions.**
   `feat(daemon): MCP server stdio loop with 22 UCIL tool descriptors`
   lands `crates/ucil-daemon/src/server.rs` (669 LOC) plus 5 lines of
   `lib.rs` re-export churn in one commit. DEC-0005 sanctions a single
   coherent-module-file commit for the daemon crate — and the
   WO's own `lessons_applied[1]` explicitly names "~300–500 LOC" as the
   comfortable ceiling; this commit is ~35% past that ceiling. The
   test body (`test_all_22_tools_registered`, ~160 LOC) could have
   landed as a separate `test(daemon): …` commit once the module was
   committable. Not blocking (const array + dispatcher + stub test
   form one build-green unit and cannot be split without a temporary
   compile break) but noted so the same pattern does not grow further
   WOs.

2. **`assert!(mock.exists(), …)` in the acceptance test is a
   guard-rail, not a weak assertion — but it can hide a broken
   harness.**
   `crates/ucil-daemon/src/plugin_manager.rs:755-759` checks the
   `mock-mcp-plugin` binary on disk before exercising the lifecycle.
   The assertion is concrete (it carries a tempdir-style diagnostic)
   and the binary genuinely needs to exist for the test to be
   meaningful — so this is not a `assert!(true)` smell. However,
   `[[bin]] name = "mock-mcp-plugin"` is declared in
   `crates/ucil-daemon/Cargo.toml:15-17` and **not** exposed via
   `CARGO_BIN_EXE_*` to library unit tests (only to integration
   tests). Verifier should ensure the cold `cargo clean && cargo
   nextest run -p ucil-daemon` invocation first compiles the bin — a
   bare `cargo nextest run plugin_manager::test_hot_cold_lifecycle`
   filtered-only run can skip building the bin and trip this guard
   with a confusing "expected mock-mcp-plugin binary at …" message.
   The ready-for-review marker (line 104–106) already lists
   `cargo nextest run -p ucil-daemon --no-fail-fast` as the verified
   command, which builds the bin; flagging so the verifier uses the
   full-crate form rather than a filter-only form.

3. **Branch is behind `main` by commit `0856177` (escalation file).**
   `git diff --name-status main..HEAD` reports a **deletion** of
   `ucil-build/escalations/20260417-0438-monitor-session-post-harness-fix.md`.
   This is a rebase artefact: the branch was cut at `e74557e`; commit
   `0856177` (`chore(escalation): monitor session post-harness-fix —
   admin, Bucket A`) landed on `main` afterwards. A plain
   fast-forward merge / rebase will resolve it automatically (the
   file stays on `main`, the branch does not modify it). If the
   verifier performs a `git merge feat/... main`-style workflow, the
   deletion will not replay. Not blocking, but the verifier should
   confirm the escalation file is still on `main` after the merge.
