### Warnings (should fix or be aware of before gate)

#### W-1. First commit `ffd3cf2` is +542 LOC — exceeds DEC-0005 ~200-LOC module-coherence soft cap and the 297-LOC WO-0048 framing precedent

`git show --stat ffd3cf2 --format=''` reports `crates/ucil-core/src/fusion.rs | 541 +++` plus `crates/ucil-core/src/lib.rs | 1 +` = **+542 LOC in one commit**.

Root `CLAUDE.md` § "Commit + push cadence (mandatory)" says:
> Commit at every logical step — new function, new test, new module, new fixture. Max ~50 lines of diff per commit.

The WO planner's `plan_summary` explicitly recommended a 3-way split inside `fusion.rs`:
> (one for new types + their derives + module declaration in lib.rs, one for `fuse_g2_rrf` + `rrf_weight` + the tracing instrument, one for the acceptance test + lib.rs re-exports + verify script)

The executor instead consolidated all three layers (types + impl + 196-LOC frozen test) into commit `ffd3cf2` and split the remaining work (re-exports, verify script, RFR marker) into 3 small commits. Total commit count is 4, satisfying AC27's "at least 3 commits" floor, but the first commit's size is **~80% over the WO-0048 297-LOC precedent** that the WO itself cited as the upper bound.

**Mitigating factors**:
1. `crates/ucil-core/src/fusion.rs` is a NEW module — splitting types from impl would have triggered `#![deny(warnings)]` cascades on unused types / unreferenced const-fns at the type-only and impl-only intermediate states. The 3-way split would require `#[allow(dead_code)]` scaffolding the executor would need to remove later.
2. The file is module-coherent — every line belongs to the same logical unit (the G2 RRF surface). DEC-0005 explicitly permits "module-coherence commits up to ~200 LOC"; the executor stretched this to 542 because the module-coherence cap is the only natural bracket.
3. AC27's hard floor (`at least 3 commits`) is met (4 commits total).

**Recommendation**: the verifier MAY accept the cadence as-is given (1) the module-coherence justification, (2) the `#![deny(warnings)]` constraint on intermediate splits, and (3) the precedent set by WO-0046 (252 LOC) and WO-0048 (297 LOC) that the WO itself cited as upper bounds. Alternatively, the verifier MAY request a rebase that splits the first commit into three (types-only with `#[allow(dead_code)]`, impl, test) — but this is a soft commit-hygiene preference, not a behavioural defect.

This is a Warning, not a Blocker, because every other commit-hygiene rule (Conventional-Commits subject, Phase/Feature/WO trailers, no `--amend` after push, no force-push, linear history) is satisfied.
