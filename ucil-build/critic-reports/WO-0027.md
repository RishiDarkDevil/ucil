2. **Watchman async forwarder IO awaits lack `tokio::time::timeout`** — `crates/ucil-daemon/src/watcher.rs:676` `stdin.write_all(subscribe_cmd.as_bytes()).await`, `:679` `stdin.write_all(b"\n").await`, `:714` `forwarder_tx.send(file_event).await`. The `.claude/rules/rust-style.md` rule ("Every `.await` that touches IO is wrapped in `tokio::time::timeout` with a named const") is not strictly observed here. `lines.next_line().await` at `:686` is a subscription stream and is legitimately infinite — acceptable. `kill_on_drop(true)` at `:655` mitigates the write/send hang risk because dropping `FileWatcher` tears the child down, but the writes/sends themselves could still stall the forwarder task if the subscribe handshake stalls the child before drop. Consider adding a named `WATCHMAN_HANDSHAKE_TIMEOUT` const + `tokio::time::timeout` around the two stdin writes.

3. **`WatcherError::WatchmanDecode(serde_json::Error)` is declared at `watcher.rs:362` but never constructed.** The Watchman forwarder handles decode failure inline via `tracing::debug!` + `continue` at `:689–700` rather than surfacing the variant. The rustdoc acknowledges the variant is reserved for future use, but exposing a public error variant that no code path emits is misleading to downstream `match` authors. Either emit it on the initial-subscribe-line decode path or drop it until load-bearing. `#[non_exhaustive]` on the enum means addition later is non-breaking, so deferring is cheap. (Was explicitly requested in WO-0027 `scope_in`, so not a blocker — accepting as reviewer-level guidance.)
