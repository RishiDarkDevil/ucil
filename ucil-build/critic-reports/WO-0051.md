## Notes for the verifier

- **DEC-0009 divergence is correct, not a smell.** The new test is parse-only; calling `PluginManager::health_check` against the ripgrep manifest would spawn `rg --version` which exits without speaking JSON-RPC and the health-check would (correctly) fail. The leading `//` comment block at `tests/plugin_manifests.rs:150-157` documents this explicitly. Per DEC-0009 (search-code-in-process-ripgrep), ripgrep is invoked in-process via `crates/ucil-daemon/src/text_search.rs` (WO-0035) — the manifest is declarative metadata so the planning layer can enumerate ripgrep alongside ast-grep + probe under the `search` category.
- **Sentinel `[transport]` table is intentional.** `command = "rg"`, `args = ["--version"]` parse cleanly through `PluginManifest::from_path` but are never spawned; if a future code path accidentally tried to drive this transport as an MCP server, `rg --version` would exit immediately and the spawn would fail loudly — desired behaviour, not a bug.
- **Workspace warm-up before `.gitignore` sub-check is intentional.** `scripts/verify/P2-W7-F07.sh:99` runs `cargo build --workspace --quiet` before the `rg --files-with-matches` invocation so `target/` exists on disk; otherwise the assertion would be vacuous (ripgrep cannot fail to descend into a directory that does not exist). On subsequent runs the build is a fast no-op against the cached `target/`.
- **Single soft warning above** does not gate the verifier — the `install-ripgrep.sh` cosmetic divergence does not affect any acceptance criterion (the script always `exit 0`s and is informational scaffolding). Verifier may proceed.
