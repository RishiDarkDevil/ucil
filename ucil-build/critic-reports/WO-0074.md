   - The body cites the WO-0072 `P3-W9-F08.sh` shape (subprocess + JSON-RPC + python3 validation) and notes the python harness is in-script for precise wall-time measurement, but does not invoke the DEC-0005 module-coherence carve-out by name.

4. **F02 verify script uses bash sleep-based JSON-RPC timing.** `scripts/verify/P3-W10-F02.sh:132-155`
   - Sends `initialize` → `sleep 1` → `notifications/initialized` → `sleep 1` → `tools/call` → `sleep 10` → close stdin → `wait $SERVER_PID`.
   - The 10-second post-`tools/call` sleep is a heuristic for letting the resolve-library-id response land in the FIFO before stdin close. On slow network (Context7 reaches context7.com over HTTPS), a >10s response wall-time would race the stdin close, drop the response on the floor, and produce a "missing JSON-RPC response for id 2" failure.
   - `wait $SERVER_PID` mitigates this since the server should flush pending responses before exiting on EOF — but the assumption depends on Context7's stdio shutdown behaviour. The F03 script uses python polling with a 60-second deadline which is more robust. Soft warning — verifier should re-run on transient failure to rule out network flake before treating it as a regression.
