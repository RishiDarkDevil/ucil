#### W-1. Upstream `scip-rust v0.0.5` is broken — install hint produces a non-working binary

`scripts/devtools/install-scip-rust.sh:31` prints:

```
cargo install --git https://github.com/sourcegraph/scip-rust --tag v0.0.5 scip-rust
```

The executor's own self-report (`ucil-build/work-orders/0055-ready-for-review.md:124`, "Notable judgment call #4") admits: **"upstream `scip-rust v0.0.5` is a 1-line `rust-analyzer scip . > dump.scip` shell wrapper that does NOT support the `--output` flag the work-order's CLI shape uses"**.  The local test passes only because the executor installed a UCIL shim at `~/.local/bin/scip-rust` that translates `scip-rust index --output <path>` into `rust-analyzer scip <cwd> --output <path>`.

Consequences for the verifier:

1. A clean-machine verifier following `scripts/devtools/install-scip-rust.sh`'s hint literally will get the broken upstream and `crates/ucil-daemon/src/scip.rs:281-291` (`scip-rust index --output ...` spawn) will fail SA1 with `IndexerExitCode { code: <non-zero>, stderr: ... }`.
2. The plugin manifest's `[indexer].binary = "scip-rust"` (`plugins/structural/scip/plugin.toml:66`) and `[indexer].args = ["index", "--output", "{output}"]` (`:68`) reflect the work-order's *intended* CLI shape, not what the pinned upstream actually accepts.
3. `scripts/verify/P2-W7-F08.sh:60` runs `(cd tests/fixtures/rust-project && scip-rust index --output "${SCIP_OUT_FILE}")` against the fixture — this will also fail on the broken upstream.

**Recommendation**: the verifier MUST either replicate the executor's shim (a one-liner that exec's `rust-analyzer scip "$PWD" --output "$3"`) or escalate via DEC-followup ADR for an alternative pinned upstream. The escalation path is precedented by DEC-0014 §"Revisit trigger" item 2 ("scip Rust crate becomes unmaintained").

This is borderline-blocker territory; the verifier is informed by the ready-for-review note, so I downgrade to a high-priority Warning.

**Recommendation**: the verifier should run `bash scripts/verify/P2-W7-F08.sh` end-to-end rather than the cargo selector in isolation. #### W-4. `[indexer]` / `[ingest]` tables in scip plugin.toml are silently dropped at parse time

The journey: commit `50d9555` added `pub indexer: Option<IndexerSection>` + `pub ingest: Option<IngestSection>` to `PluginManifest` per WO scope_in line. Commit `2ff938b` reverted those fields because `tests/integration/test_plugin_lifecycle.rs:111` and `:137` (in `forbidden_paths`) construct `PluginManifest { ... }` via struct literal, which Rust requires to initialize every field.

Net effect: `IndexerSection` + `IngestSection` types ride on at `crates/ucil-daemon/src/plugin_manager.rs:264-309` as documentation only; the `[indexer]` and `[ingest]` tables in `plugins/structural/scip/plugin.toml:65-78` are silently dropped at parse time (no `#[serde(deny_unknown_fields)]` ⇒ unknown tables tolerated). `PluginManifest::from_path` accepts the manifest cleanly, satisfying acceptance line 53. The owner module (`crates/ucil-daemon/src/scip.rs`) hard-codes the binary name and schema, bypassing the manifest entirely.

**Severity**: low. Behaviorally OK for this WO because the SCIP module owns its own invocation. But the pre-baked `[indexer]` / `[ingest]` schema is currently dead-letter — a future plugin loader that wants to spawn indexers from the manifest will need to re-introduce the fields (or re-parse via a sidecar struct as commit `2ff938b`'s body suggests). The executor's ready-for-review note #2 documents this trade-off explicitly.
