### Warnings (should fix before gate — non-blocking)

1. **`std::process::Command` inside async helpers**
   `crates/ucil-daemon/tests/g6_plugin_manifests.rs:70` imports
   `std::process::Command`; helpers `assert_is_git_repo`
   (`tests/g6_plugin_manifests.rs:182-196`) and
   `init_git_repo_if_needed`
   (`tests/g6_plugin_manifests.rs:203-250`) call `.output()` on it.
   The helpers are invoked from the async test body
   (`git_manifest_health_check` at
   `tests/g6_plugin_manifests.rs:332` — see line 365-366) so each
   `git init` / `git add` / `git commit` blocks the tokio worker.
   `.claude/rules/rust-style.md` §"Async" line 23 mandates
   `tokio::process::Command` in async paths. The blocking footprint
   is small (single fixture init, ≤300 ms total), and the prior
   `g3/g4/g5_plugin_manifests.rs` peer files do not exercise this
   shape (no git-repo fabrication needed there), so there is no
   established precedent to override the rule here. Mitigation
   would be a one-line switch to `tokio::process::Command` with
   `.output().await`. Soft warning — not gate-blocking but worth
   following up.
