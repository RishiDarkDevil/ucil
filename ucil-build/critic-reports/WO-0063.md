### Warnings (should be acknowledged before gate)

1. **`G2_MASTER_DEADLINE` is declared but never enforced** — `crates/ucil-daemon/src/server.rs:107` declares `pub const G2_MASTER_DEADLINE: Duration = Duration::from_secs(5)` with rustdoc claiming "Bounds the total wall-time across all parallel providers — the outer cap on top of the per-source deadlines", but `run_g2_fan_out` (`server.rs:1473-1517`) does NOT wrap its `for task in tasks { task.await }` join loop in `tokio::time::timeout(G2_MASTER_DEADLINE, ...)`. A pathological stack of three providers each timing out at the per-source deadline (2 s) would still respect the 5-s master cap because they run in parallel via `tokio::spawn`, but if any provider's `tokio::spawn` task hung past its inner timeout (e.g. `JoinError` / a future not yielding), the outer fan-out has no escape hatch. The WO scope_in step 3 prescribes both consts but only explicitly wires the per-source one. **Recommendation**: either (a) wrap the for-loop in `tokio::time::timeout(G2_MASTER_DEADLINE, ...)` and on `Elapsed` log a `tracing::warn!` and proceed with whatever per-source results have been collected so far, or (b) downgrade `G2_MASTER_DEADLINE`'s rustdoc to "documentary only — the per-source deadline + parallel fan-out provide the effective bound" so future readers don't expect timeout semantics that aren't there. Not a blocker because the per-source deadline + `tokio::spawn`'s parallelism keep the practical worst-case at ~2 s, but the unused public const is visible API surface that future maintainers will reasonably read as enforced.
