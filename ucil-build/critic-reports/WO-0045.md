### Warnings (should fix before gate, non-blocking for verifier)

1. **Stale doc comment on `PluginArgs::command`** —
   `crates/ucil-cli/src/commands/plugin.rs:54-56` still says
   *"Today only `install` exists; `list`, `enable`, `disable`,
   `uninstall`, `reload` are reserved for later work-orders per
   master-plan §16."* That sentence is now factually wrong — every
   subcommand it names is shipped in this WO. The next executor in
   this file (or a docs-writer pass) should rewrite it to describe
   the full six-variant tree.

2. **Stale module-level / `OutputFormat` doc** —
   `crates/ucil-cli/src/commands/plugin.rs:1-20` is the original
   `install`-only module preamble, and `OutputFormat`'s rustdoc
   (`plugin.rs:35`) calls itself "Output format for the `plugin
   install` subcommand" even though it now backs all six. Same
   doc-rot pattern as #1; cosmetic, but the file's surface no
   longer matches its top-of-file claim.

3. **Mutation-check type-name drift vs WO acceptance text** —
   The WO `acceptance[24..26]` describes the mutation patches as
   `Ok(UninstallOutcome::default())` / `Ok(EnableOutcome::default())`,
   but the executor factored `uninstall`/`enable`/`disable` through a
   shared `StateChangeOutcome` (`plugin.rs:546`) which does **not**
   derive `Default` (it has a `&'static str` field).
   `0045-ready-for-review.md:107-116` acknowledges this and supplies
   the literal-construction patch
   `Ok(StateChangeOutcome { entry: PluginStateEntry::default(),
   status_label: "uninstalled" })`. The verifier should patch the
   bodies with that explicit construction rather than `::default()`.
   `ReloadOutcome` does derive `Default`, so the third mutation
   target works as written.

4. **`0045-ready-for-review.md:108` mis-describes the failure line of
   mutation #1** — for `test_plugin_uninstall_marks_state_file`, the
   test pre-seeds the state file (`plugin.rs:1173-1184`), so a
   no-op `uninstall_plugin` body does **not** trip
   `read_state_entry_blocking(...).expect("state row present after
   mutation")` (the row is still present from the seed). The panic
   actually fires on `assert!(!state.installed, ...)` at
   `plugin.rs:1201-1204`, because the seeded `installed=true` is
   never flipped. Functionally the mutation check is still sound
   (the test fails on the correct mutation), but the verifier
   shouldn't be misled by the misattributed assertion line. The
   `enable` and `reload` lines in that note are accurate.

5. **Commit-size soft-target overruns** — 4 of 7 feature commits
   exceed the ~50-LOC soft target from
   `.claude/rules/commit-style.md`:
     - `d19e0dc feat(cli): add PluginStateEntry + read/write/mutate state helpers` — 190 LOC
     - `581e9be feat(cli): add 'plugin list' subcommand with --format json` — 132 LOC
     - `11cbe83 feat(cli): add 'plugin uninstall|enable|disable' state-mutating subcommands` — 178 LOC
     - `01f334e feat(cli): add 'plugin reload' subcommand wrapping health_check_with_timeout` — 136 LOC
   `DEC-0005` covers only the test commit (`598f91e`, 409 LOC).
   Each feature commit is one coherent change, so this is a soft-
   target observation, not a violation. Worth flagging since prior
   WOs (e.g. WO-0042/0043) split similar work into 2–3 smaller
   feature commits each.

6. **`PluginArgs.command` field doc placement** —
   `crates/ucil-cli/src/commands/plugin.rs:53-57` puts the doc
   comment **after** the `#[command(subcommand)]` attribute, which
   is the same placement as the pre-existing code. clippy /
   rustdoc accept it, but it's stylistically inconsistent with the
   rest of the file. Pre-existing — only flagged because the
   stale-doc fix in #1 is the natural moment to move it above the
   attribute.
