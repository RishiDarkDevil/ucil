**2. F11 acceptance criterion passes trivially with 0 tests run**

The work order states:
```
cargo test -p ucil-core -- --ignored fixture::rust_project_loads exits 0
```

The `--ignored` flag only executes tests carrying `#[ignore]`. The test at
`crates/ucil-core/tests/fixture.rs:16` is declared as:

```rust
#[test]
pub fn rust_project_loads() { … }
```

It has **no `#[ignore]` attribute**. Running `cargo test -p ucil-core -- --ignored …`
will therefore match 0 tests and exit 0 without exercising any assertion. The real
cargo-check sub-process is hidden behind `UCIL_SLOW_TESTS=1`. The commit message
acknowledges this trade-off:

> "the slow cargo check is gated behind UCIL_SLOW_TESTS=1 to satisfy the pre-commit
> hook that blocks bare #[ignore]"

This creates a gap: the stated acceptance criterion is vacuously satisfied while proving
nothing. The executor must resolve the conflict between the work-order acceptance criterion
and the pre-commit hook before the verifier is spawned. Options:

  a. Add `#[ignore]` to `rust_project_loads` and update the acceptance criterion to use
     `UCIL_SLOW_TESTS=1 cargo test -p ucil-core --test fixture` (requires a hook ADR or
     whitelist entry for "slow" `#[ignore]` tests that are not masking failures).
  b. Keep the env-var gate but update the acceptance criterion to:
     `UCIL_SLOW_TESTS=1 cargo test -p ucil-core --test fixture fixture::rust_project_loads`
     and file an ADR noting the deviation from the work-order spec.
  c. Remove the env-var gate entirely and always run `cargo check` (slowest, keeps the
     acceptance criterion as written, but requires either allowing `#[ignore]` or
     accepting a ~30-second test).

Until one of these is resolved the criterion is a false green.

`crates/ucil-core/tests/fixture.rs:~44` uses `std::process::Command::new("cargo")`.

> "All shell invocations use `tokio::process::Command`, never `std::process::Command`

The test is non-async (no `#[tokio::test]`), so there is no strict violation; however,
if this test is later made async, the rule would require a refactor. Note this now rather
than inherit it.
