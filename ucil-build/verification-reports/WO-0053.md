## Reproductions

### AC17 — fail

```bash
$ cd /home/rishidarkdevil/Desktop/ucil-wt/WO-0053
$ ! grep -nE '#\[ignore\]|todo!\(|unimplemented!\(|//[[:space:]]*assert' \
    crates/ucil-daemon/src/branch_manager.rs
238:    /// assert_eq!(mgr.branches_root(), root.as_path());
239:    /// assert_eq!(mgr.archive_root(), root.join(ARCHIVE_DIR_NAME).as_path());
280:    /// assert_eq!(
399:    /// assert_eq!(info.branch, "main");
400:    /// assert!(info.vectors_dir.ends_with("main/vectors"));
492:    /// assert!(archived_at.starts_with("/repo/.ucil/branches/.archive"));
$ echo "exit=$?"
exit=1
```

The regex's third alternative `//[[:space:]]*assert` is intended to flag
commented-out test assertions (e.g. `// assert!(...)`). It matches `///   assert`
because GNU grep's `//` matches positions 5–6 of `    /// assert_eq!`, then
`[[:space:]]*` consumes the third `/` (no — wait, `/` is not whitespace; the
match actually starts at columns 5–6 with `//` matching the leading two
slashes, then `[[:space:]]*` matches zero whitespace, then `assert` would
need to immediately follow… let me re-check). The actual greedy ERE
behaviour: GNU grep finds `//` at position 5 (the first two of three
slashes), then `[[:space:]]*` matches zero or more whitespace (zero is
allowed), then it needs `assert` immediately after. Position 7 is `/`
(third slash), not `assert`, so why does it match? Answer: ERE is greedy
but tries multiple anchor positions. Position 6 (second + third slash) is
also `//` — so grep slides to that position, then `[[:space:]]*` consumes
the leading spaces of `   assert`, then `assert` matches. Either way,
the regex matches `///   assert…` lines.
