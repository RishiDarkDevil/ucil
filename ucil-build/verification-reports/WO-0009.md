### P1-W3-F04 — script-level zero-tests false-positive, manually verified

`scripts/reality-check.sh P1-W3-F04` exits **1** with message
> `FAILURE: acceptance tests reported ZERO tests run with code stashed — module was removed, not a genuine pass.`

This is the documented single-commit new-module harness false-positive
the orchestrator has already accepted twice — `verifier-bb3cb69d` for
`P1-W2-F01` and `P1-W2-F05` (WO-0005, commit `bec3f35`) — and tracked in
escalation `20260415-1630-reality-check-per-file-rollback.md` (marked
`resolved: true`). When the whole feature module is introduced in a
single commit, stashing every changed source file leaves the crate
without the module the tests select, so nextest's "no tests ran" path
fires before the tests themselves get to fail. No rollback-based script
can distinguish "module never existed" from "module exists but is broken
enough to compile-but-not-match" at that commit; the fundamental gap is
structural. The semantic invariant (removing the feature code prevents
the acceptance tests from passing) is still satisfied — and I verified it
directly in this session:
