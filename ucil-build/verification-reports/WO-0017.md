## Harness bugs observed (non-blocking, for triage Bucket B)

1. `scripts/reality-check.sh` `git stash pop` at line 190/200 pops the top-of-stack
   stash even when no stash was created by the immediately-preceding
   `git stash push`. When stale `auto-stash-on-resume-*` entries exist (cross-worktree),
   the pop applies someone else's changes, corrupts `Cargo.toml`, and trips a
   false-negative `FAILURE: tests fail with code restored`. Suggested fix: tag the
   push with `-m "reality-check-$FEATURE_ID"` and only pop by that message (e.g.
   `git stash pop "$(git stash list | grep -m1 'reality-check-$FEATURE_ID' | cut -d: -f1)"`).
2. The executor's deliberate omission of the `Feature: P1-W2-F02` trailer from
   four `test(…)` commits (`3f29a43`, `472e2ce`, `aceafc3`, `7d7314d`) compensates
   for `scripts/reality-check.sh`'s plain-substring `git log --grep='Feature: $ID'`,
   which would otherwise union `tests/` survival files into the stash set and
   defeat the oracle. Suggested fix: use `git interpret-trailers` or
   `git log --grep='^Feature: $ID$' --extended-regexp` anchored to trailer lines.
3. The repo-global stash stack from prior-worktree `auto-stash-on-resume-*`
   cleanups should be pruned by `scripts/resume.sh` when the targeting branch
   has no dirty files, not carried forward indefinitely.

These are harness hygiene issues, not P1-W2-F02 defects, and should be
escalated as a Bucket-B micro-WO (matches critic report WO-0017.md warnings 2-4).
