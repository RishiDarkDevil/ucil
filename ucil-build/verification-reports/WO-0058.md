### Coverage gate diagnosis

Cached non-instrumented test binaries from the prior workspace-wide AC11 run were reused, producing zero `.profraw` output, which then breaks the report step.
- After running `cargo clean` to invalidate the cache, the same `coverage-gate.sh ucil-embeddings 85 75` invocation succeeds: `[coverage-gate] PASS — ucil-embeddings line=88% branch=n/a`.
- The actual coverage was confirmed with the atomic form `cargo llvm-cov --package ucil-embeddings --summary-only --json` (which `cargo-llvm-cov` 0.8.5 manages atomically and which always works regardless of cache state): line 87.80%, region 86.61%, function 70%.

**Verdict**: gate exits 0 after `cargo clean`. Coverage report at `ucil-build/verification-reports/coverage-ucil-embeddings.md` regenerated by the script with `Verdict: PASS`.

The cache-invalidation sensitivity is a known harness ergonomic — verifiers that run AC11 (workspace-wide test) before the coverage gate must `cargo clean` between them. This is not a script bug per se (cargo-llvm-cov 0.8.5 documents the same target-dir-sharing constraint upstream) but is a verification-report-worthy footnote for the next harness-fixer pass.
