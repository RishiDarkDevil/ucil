LanceDB+IVF/HNSW vector-query-latency bench (master-plan §18 Phase 2 Week 8
line 1789). The implementation comprises (1) four `[dev-dependencies]`
(`lancedb`, `arrow-array`, `arrow-schema` workspace pins + `rand = "0.8"`

(`CORPUS_SIZE = 2000`, `QUERY_COUNT = 20`, `EMBEDDING_DIM = 768`), a

the `embedding` column with `num_partitions = ceil(sqrt(2000)) = 45`, and
a `bench_vector_query_p95` criterion function with `sample_size(100)`,
`warm_up_time(2s)`, `measurement_time(20s)`; (3) `scripts/bench-vector-query.sh`
(162 LOC) — release build cache warm + `cargo bench` + jq-driven per-sample
p95 computation + dual asserter (wall-time floor `MEAN_NS_PER_ITER >= 100_000` +
p95 floor `< 100 ms`) + canonical `p95_warm_ms=<N>` stdout; (4)
`scripts/verify/P2-W8-F07.sh` (90 LOC) — frozen-script wrapper that runs
the bench script and INDEPENDENTLY re-asserts `< 100 ms`.

End-to-end measurements on this verifier host (clean per-crate cargo +
warm cache):
- `bash scripts/bench-vector-query.sh` → `p95_warm_ms=0.276` → `[OK]`
- `bash scripts/verify/P2-W8-F07.sh` → `[OK] P2-W8-F07 p95_warm_ms=0.271`

Both readings are ~3 orders of magnitude below the master-plan §18 line 1789
floor of 100 ms. Independent jq-recompute of `target/criterion/.../sample.json`
on this host: `n=100, mean_ns_per_iter=259093, p95_ns=270857, p95_ms=0.271` —
matches the bench script's reading within noise. Wall-time floor (100 µs /
100 000 ns per iter) is comfortably cleared at ~259 µs/iter mean.

**Mutation contract caveat (documented divergence on AC31)**: Mutation M2
(`QUERY_COUNT` shrunk to 1) did NOT empirically breach the wall-time floor
as the WO predicted. LanceDB+HNSW per-query latency at corpus N=2000 is
independent of query identity — there is no aggressive per-query result
cache to exploit at this corpus size. The executor's RFR mutation log
explicitly hedged this ("Note: depending on LanceDB's caching behaviour
the mean may stay above 100 µs but the p95 still drops markedly"); empirically
neither mean nor p95 dropped meaningfully (mean=254367 vs 259093; p95=270630
vs 270857 — both within statistical noise). The wall-time floor's load-bearing
job is M1 (no-op nearest_to body), which DID breach the floor decisively
(MEAN_NS_PER_ITER=0.9166 ns ≪ 100 000). M3 is verified by inspection per
the WO's own acceptance text. The bench is real, the safety net works, and
the substantive verification (real bench, real measurements, real assertions,
all functional ACs green, phase-1 gate green) is solid. M2 is a non-attack
against the LanceDB+HNSW path and the WO's prediction was a contract-design
defect, not a bench defect.
