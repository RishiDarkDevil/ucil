### Note on `scripts/reality-check.sh`

The verifier ran `scripts/reality-check.sh P2-W6-F07` per workflow step 6b. The script exited 1 with `FAILURE: acceptance tests PASSED with code stashed`, which on its face is a fake-green signal. **Investigation determined this is a false positive caused by a script-methodology limitation**, not a fake-green code path:

- The feature spans 7 commits on `feat/WO-0045-...`, with the test-introducing commit (`598f91e`) being the topmost source-touching commit on the feature's git-trailer-grep `git log --grep="Feature: P2-W6-F07"` list.
- For each changed file, reality-check picks `_file_commit[$f] = first matching commit (newest-first)` and rolls the file back to that commit's parent.
- For `crates/ucil-cli/src/commands/plugin.rs`, that's `598f91e^` — i.e. the file state with all five new handlers (`list_plugins`, `uninstall_plugin`, etc.) intact but with the new tests stripped.
- Running `cargo test -p ucil-cli commands::plugin::` against that rollback then runs only the **pre-existing** install + emit-helper tests (which were green before this WO) and exits 0.
- The script interprets exit 0 as "tests passed with code stashed → fake-green", but in fact the new tests aren't even in the binary at that point — they were rolled back along with the test commit.

This limitation has been documented in prior verifier reports (e.g. WO-0044 cited the same workaround chain). The five per-handler targeted mutations above are the **authoritative anti-laziness layer** for this WO and prove the new tests genuinely exercise each new code path. DEC-0007 explicitly notes that reality-check + critic stub-scan + coverage gate are the per-WO layers, and the WO planner pre-baked the targeted mutations into `acceptance[24..26]` to handle exactly this kind of multi-commit feature.
